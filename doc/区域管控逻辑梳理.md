# 逻辑梳理
中间使用的ip地址均为模拟环境，可修改
区域管控逻辑基本与风淋门逻辑一致，对rcs来说还是风淋门。
区别是对接的ACS系统的报文不同

对应rcs上配置两个门编号使用q001和q002（默认，可自定义修改配置）
两个门需要本地缓存记录状态

整理逻辑不变，但是如果查询到区域内有其他设备不让进，那么不会发送请求开门，只会不停查询状态等待区域释放后再请求开门。

## 区域管控逻辑

### 区域查询逻辑

- 定时轮询查询，约5分钟一次（可配置）
- 请求进入区域前后及出区域前后均需要查询（确保信息实时性）后续请求逻辑自行添加，不额外添加

> curl --location --request GET 'http://10.68.2.10:5012/api/zones/status?zone_id=zone_001'

请求
```
{
  "permission_id": "perm_123456",//可配置,许可证
  "zone_id": "zone_001",//可配置
  "client_id": "iray",//可配置
  "status": "granted"//可配置
}
```
响应示例（已占用、不可用）
```
{
  "zone_id": "zone_001",
  "status": "occupied",//使用中的
  "occupied_by": "client_b"
}
```
响应示例（可用、可用）
```
{
  "zone_id": "zone_001",
  "status": "available"//可使用的
}
```



### 请求逻辑
1. AGV rcs 请求开门

    > http://10.68.2.10:7110/changeDoorStatus
    {
    "doorNum": "q001",      // 门编号
    "doorStatus": "1"       // 1=开门, 2=关门
    }

2. ASAP 转换发送请求进入区域（post）和查询状态（get），根据api文档，一直发送

    > http://10.68.2.10:5012/sim/api/zones/enter
{
  "zone_id": "zone_001",
  "client_id": "iray"
}


    直到报文查询到区域空闲可用（区域查询请求，一直查询，查不到就一直查）

3. 主动上报rcs开门到位状态
    > http://10.68.2.10:7110/changeDoorStatus
    {
    "doorNum": "q001",      // 门编号
    "doorStatus": "1"       // 1=开门, 2=关门
    }

4. AGV 过门后rcs会下发控制关门报文，这一块好像不需要额外处理，只需要过门后将状态改为关门

    rcs 会发送控制关门的请求，正常响应并将本地缓存q001门置为关门即可

5. 持续获取区域状态，等待rcs发送q002关门请求，将本地缓存的q002门状态置为关门，然后发送离开区域的请求。持续发送查询请求匹配字段判断确认是否已从不可用变更为可用，如果有出现丢包情况则尝试重发离开区域

    > http://10.68.2.10:5012/sim/api/zones/exit
    {
        "zone_id": "zone_001",
        "client_id": "iray"
    }


这是一个单线程，意味着两个门不可能同时打开，只有在第一个门关闭后才能打开第二个门。例如从q001门去q002，必须等q001门关了之后q002才能请求离开区域。反向同理。
