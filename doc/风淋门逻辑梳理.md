# 逻辑梳理
中间使用的ip地址均为模拟环境，可修改

风淋门对应rcs上配置的门编号使用1001和1002（默认，可自定义修改配置）

现在梳理一下关于报文中的“进”和“出”要怎么判断，从左往右行驶是出，从右往左行驶是进，也就是车从1001进入，然后再从1002出门，这个算是出，从1002进入再从1001出门，这个算是进。

设计思路，对rcs来说，就是车到了哪里，进请求开哪个门，但是会有一个先后顺序，也就是判断进出的逻辑，1001和1002都关的情况下，先给1001发开门就是出，先给1002发开门，就是进。对于获取门状态和控制门状态直接转换后发给ACS，等获取到对应状态后再回给rcs进行，也不用什么复杂逻辑。

不用考虑什么风淋等待，就一直发开门请求就行。本质上就是rcs发什么，我们转换后发过去，然后那边回什么，翻译一下再回给rcs

整个链路是单向的，就是rcs请求，经过转换获取后返回给rcs需要的报文。

## 问题

### 1. RCS 下发开门请求后，ASAP 应该：

(B) 阻塞等待 ACS 返回 doorStatus: "1" 后再返回

### 2. Direction 状态持续性问题

判定后，这个 Direction 要写入后续所有 ACS 请求（包括关门），直到两门全关后才重置

### 3. 并发、互斥

RCS 同时对 1001 和 1002 发开门请求那就同时发，只做翻译转换工作

### 4. Door Code ↔ Direction 映射表

情况|	1001→DOOR01 的 Direction	|1002→DOOR02 的 Direction
-|-|-
初始态（两门都关）先开 1001|	"2" (出)	|"2" (出)
初始态（两门都关）先开 1002|	"1" (进)	|"1" (进)

## 风淋门逻辑

### 一、AGV rcs查询门状态接口（RCS端主动查询）

RCS 端
        > http://10.68.2.10:5012/api/rcs/doorStatus
    **第三方系统需要提供 HTTP 接口**，URL 配置在数据库 `access_config.door_status_relation_id` 字段。

    **调用方式：** `HTTP POST`

    ```json
    // 请求体
    {
    "doorCode": "1001"
    }

    // 成功响应
    {
    "code": 1000,
    "msg": "success",
    "data": {
        "status": 1      // 0=离线, 1=开门, 2=关门
    }
    }

    // 失败响应（本系统会将该门标记为 OFFLINE）
    {
    "code": 9999,
    "msg": "error",
    "data": null
    }
    ```

需要转换成的

> http://10.68.2.10:5012/sim/acs/door/DOOR01
**调用方式：** `HTTP GET`
```json
{
    "doorSerial": "DOOR01",
    "doorStatus": "1",
    "command": "1",
    "code": "200"
}
```

doorStatus|String门系统状态ACS
-|-
0|已关闭
1|完全打开到位
2|故障

需要翻译成rcs对应的状态，即这里获取到的0对应rcs中的2,1对应rcs中的1，2对应rcs中的0

### 二、 AGV rcs请求开门及关门 （RCS端主动请求开门以及关门）
    rcs 请求开门和请求关门报文

    > http://10.68.2.10:5012/api/rcs/doorStatus
    {
        "deviceNum": "1746",//设备编号，标记用
        "doorCode": "1001",//门编号
        "payLoad": "0",//负载情况，没用
        "qrName": "81111096",//设备当前点位，没用
        "orderId": "",//任务id，没用
        "deviceCode": "test0",//设备序列号，设备唯一编号
        "status": 1//1表示请求开门，2表示请求关门
    }

    // 期望响应
    // {
    //   "code": 1000,               // 1000=成功, 其他=失败
    //   "msg": "success",
    //   "data": null
    // }

    // 失败响应（控制操作）
    // ACS 返回/状态          RCS 应返回
    // doorStatus: "2" (故障)  {code: 9999, msg: "door fault"}
    // HTTP 超时/不可达       {code: 9999, msg: "timeout"}
    // code: "500" (异常)     {code: 9999, msg: "door error"}

    将请求开门报文进行翻译
     ASAP 转换发送请求开门AB门和查询状态，根据api文档，一直发送

    > http://10.68.2.10:5012/sim/acs/door/DOOR01

```json
{
    "doorSerial": "DOOR01",     //自动门序列号，与请求一致
    "command": "1",             //"0"：无动作 "1"：打开 "2"：关闭
    "Direction": "1",           //"1"：进 "2"：出，根据前文逻辑判断
    "RobotName": "AGV-RCS"
}
```

直到报文返回 "doorStatus": "1",表示完全开到位，查询到非1即关门或离线，默认关门

doorStatus|String门系统状态ACS
-|-
0|已关闭
1|完全打开到位
2|故障
